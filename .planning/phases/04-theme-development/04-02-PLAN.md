**File 1: theme-json-guide.md (THM-25)**

Structure:
1. **Quick Reference Table** -- Every theme.json v3 top-level key with type, required/optional, description, and example. Organized in sections: metadata (version, $schema), configuration (settings), appearance (styles), structure (templateParts, customTemplates), content (patterns).
2. **Version History and Migration** -- v1 (WP 5.8, deprecated), v2 (WP 5.9, still works), v3 (WP 6.6, current). Breaking changes in v3: defaultFontSizes defaults to true (v2: false), defaultSpacingSizes defaults to true (v2: false). Migration steps from v2 to v3 with BAD/GOOD pairs. Flag v1 as CRITICAL, v2 as WARNING.
3. **Settings Deep Dive** -- Complete property catalog for each settings section:
   - color: palette (slug/color/name), gradients, duotone, custom, defaultPalette, defaultGradients, link, button, heading, background, text. BAD: hardcoded colors in templates. GOOD: theme.json palette with preset usage.
   - typography: fontSizes (slug/size/name/fluid), fontFamilies (fontFamily/name/slug/fontFace), customFontSize, fontWeight, letterSpacing, textDecoration, textTransform, lineHeight, fluid. Fluid typography: { "fluid": { "minFontSize": "16px", "maxViewportWidth": "1600px" } }.
   - spacing: padding, margin, blockGap, units (array of valid units), spacingSizes (custom), spacingScale (auto-generated), customSpacingSize. BAD: hardcoded padding in templates. GOOD: spacing presets.
   - layout: contentSize (default content width), wideSize (wide alignment width). Both critical for block alignment behavior.
   - border: color, radius, style, width, customColor, customRadius, customStyle, customWidth.
   - shadow: presets (slug/name/shadow), defaultPresets.
   - dimensions: aspectRatio, minHeight.
   - position: sticky.
   - appearanceTools: true shorthand (enables border, color.link, spacing, typography collectively -- WP 6.5+ default behavior).
4. **Styles Deep Dive** -- How styles cascade in theme.json:
   - Root-level styles: color (background, text, gradient), typography (fontFamily, fontSize, fontStyle, fontWeight, letterSpacing, lineHeight, textDecoration, textTransform), spacing (padding, margin, blockGap), border, shadow, outline, css (custom CSS, WP 6.2+).
   - Element styles: heading (and h1-h6), link (:hover pseudo), button, caption, cite. Selector specificity chain.
   - Block styles: per-block overrides keyed by block name ("core/paragraph", "core/group"). Full style support within each block key.
   - Precedence order: theme.json defaults < theme styles < user global styles < per-block styles.
5. **templateParts Configuration** -- Array of objects: { "name": "header", "title": "Header", "area": "header" }. Valid areas: header, footer, uncategorized (general in older docs). Name must match filename in parts/ directory (without .html). Title shown in Site Editor UI. BAD: template part file without theme.json registration. GOOD: registered with name, title, area.
6. **customTemplates Configuration** -- Array of objects: { "name": "page-no-title", "title": "Page No Title", "postTypes": ["page", "post"] }. Name must match filename in templates/ directory (without .html). postTypes restricts which post types can use the template. BAD: custom template file without theme.json registration (won't appear in editor). GOOD: registered in customTemplates.
7. **Patterns Configuration** -- Array of pattern slugs from WordPress.org pattern directory. These are REMOTE patterns bundled with the theme. Not to be confused with local patterns in patterns/ directory. BAD: patterns array with local pattern slugs (those go in patterns/ directory). GOOD: WordPress.org pattern directory slugs.
8. **Validation Rules** -- Required fields per theme type. Valid value types for each property. Deprecated properties (v1/v2 that changed in v3). Unknown property detection. Common JSON syntax errors in theme.json (trailing commas, unquoted keys). Tools: $schema URL for IDE validation, WordPress Theme Check plugin.
9. **useRootPaddingAwareAlignments (THM-20)** -- What it does: applies root padding to body and removes it for full-width aligned blocks. Required for proper full-width layouts. settings.layout.useRootPaddingAwareAlignments: true. Must pair with styles.spacing.padding on root. BAD: full-width blocks break without this setting. GOOD: enabled with root padding defined.
10. **Common theme.json Mistakes** -- Table: version as string not number, trailing commas in JSON, missing slug in palette entries, fontFamilies without fontFace (fonts won't load), conflicting settings between theme.json and add_theme_support(), palette colors without name (display issue in editor), spacingSizes without slug, setting layout without contentSize (blocks have no max width), duplicating core presets in custom palette (bloated editor UI).

All JSON examples use valid syntax with annotations via comments above the JSON object. PHP examples for functions.php interactions follow WordPress PHP Coding Standards.

**File 2: template-patterns.md (THM-26)**

Structure:
1. **Quick Reference Table** -- Template hierarchy side-by-side: Classic (.php) | Block (.html) | Purpose | Fallback. Cover all standard templates: index, front-page, home, single, page, archive, category, tag, author, date, search, 404, singular, attachment.
2. **Block Theme Template Structure** -- templates/ directory with .html files. Block markup anatomy: `<!-- wp:template-part {"slug":"header","tagName":"header"} /-->`, content blocks, footer part. Minimal valid template. Complete template with header part, post content, footer part. BAD: PHP code in .html template. GOOD: block markup only.
3. **Classic Theme Template Structure** -- PHP template files with the Template Hierarchy. The Loop (while have_posts / the_post), Template Tags (the_title, the_content, the_excerpt, the_permalink, the_post_thumbnail), get_template_part() for modular loading. BAD: duplicated HTML across templates. GOOD: get_template_part() for shared sections.
4. **Template Hierarchy Fallback Chain** -- Complete fallback visualization: category-slug.html -> category-id.html -> category.html -> archive.html -> index.html (block) and equivalent for classic. How WordPress resolves templates in both systems. Same logic, different file extensions.
5. **Template Parts Deep Dive (THM-04, THM-18)** -- parts/ directory structure. Header part (Site Title, Navigation, Site Logo blocks). Footer part (paragraphs, social links, site info). Sidebar part (widgets block for backward compatibility, custom blocks). Area designations: header (shows in Header section of Site Editor), footer (Footer section), uncategorized (general). BAD: inline header HTML in every template. GOOD: wp:template-part reference.
6. **Conditional Tags (Classic Themes, THM-06)** -- is_front_page(), is_home(), is_singular(), is_single(), is_page(), is_archive(), is_category(), is_tag(), is_author(), is_search(), is_404(). When to use each. Common confusion: is_front_page() vs is_home() (front page is static page, home is blog page). BAD: wrong conditional tag for context. GOOD: correct conditional with explanation.
7. **Custom Templates (THM-17)** -- Block themes: file in templates/ + customTemplates entry in theme.json + postTypes restriction. Classic themes: Template Name comment in PHP file header. How users select custom templates in the editor. BAD: custom template without registration (block theme). GOOD: registered in theme.json customTemplates.
8. **Template Part Areas** -- header, footer, uncategorized areas and their UI placement in Site Editor. Custom areas (not yet supported in WordPress core). How template parts appear in the Inserter by area. BAD: all parts as uncategorized (poor organization in editor). GOOD: appropriate area for each part.
9. **Block Markup in Templates** -- Core blocks used in theme templates: wp:template-part, wp:group, wp:post-title, wp:post-content, wp:post-excerpt, wp:post-date, wp:post-author, wp:post-featured-image, wp:query, wp:query-loop, wp:query-pagination, wp:site-title, wp:site-logo, wp:navigation, wp:search. Nesting and attributes. Cross-reference wp-block-development for block markup details.
10. **Anti-Patterns** -- PHP template tags in block templates (won't work), hardcoded HTML without block wrappers, missing template part references (duplicated header/footer), classic templates without escaping output (security -- cross-reference wp-security-review), mixing PHP and block markup approaches in same template.

PHP examples: WordPress PHP Coding Standards. HTML examples: valid block markup. BAD/GOOD pairs throughout. Security cross-references for classic template escaping.

**File 1: fse-guide.md (THM-25 partial)**

Structure:
1. **Quick Reference Table** -- FSE feature table: Feature | Files Involved | theme.json Section | WP Version | Description. Cover: global styles, style variations, block patterns, navigation, font management, layout system, custom CSS.
2. **Full Site Editing Overview** -- What FSE means: entire site (header, footer, sidebar, templates) editable through the block editor. Site Editor (Appearance > Editor) as the central tool. Relationship between theme.json, templates, and the Site Editor UI. How user customizations override theme defaults.
3. **Global Styles System** -- How theme.json styles map to the Global Styles UI in the Site Editor. settings control what options appear in the editor. styles set the defaults. Users can override via Global Styles panel. Precedence: theme.json < user customizations < per-block styles. How to think about theme.json as "defaults, not requirements."
4. **Style Variations (THM-19)** -- styles/ directory structure. Each file is a partial theme.json overriding settings and/or styles. Must include "title" at root level. Naming: descriptive-slug.json (e.g., dark-mode.json, sunset.json). What can be overridden: colors, typography, spacing. What cannot: layout, templateParts, customTemplates. Users select in Global Styles > Browse Styles. BAD: style variation overriding templateParts (ignored). GOOD: variation changing color palette and typography only.
5. **Block Patterns in Themes (THM-11)** -- patterns/ directory with PHP files. Required headers: `<?php /** * Title: Hero Section * Slug: mytheme/hero-section * Categories: featured, banner */`. Auto-registered by WordPress (no register_block_pattern() needed). Pattern content is PHP that outputs block markup (can use PHP for dynamic data). Pattern categories: text, media, design, featured, banner, buttons, columns, query, posts. BAD: register_block_pattern() in functions.php for theme patterns. GOOD: PHP file in patterns/ directory with proper headers.
6. **Navigation Configuration** -- wp:navigation block in header template part. Menu creation via Site Editor. Fallback behavior when no menu exists. Navigation block attributes: overlayMenu, hasIcon, icon. Classic menus (register_nav_menus) vs block navigation. BAD: hardcoded navigation HTML. GOOD: wp:navigation block.
7. **Font Management** -- theme.json settings.typography.fontFamilies with fontFace declarations. Local font hosting: fonts in assets/fonts/, fontFace src pointing to "file:./assets/fonts/font.woff2". Google Fonts vs local hosting (performance, privacy). Font Library API (WP 6.3+) for user-managed fonts. BAD: wp_enqueue_style for Google Fonts CDN. GOOD: local font hosting via theme.json fontFamilies.
8. **Layout System** -- settings.layout.contentSize and wideSize. How alignwide and alignfull use these values. useRootPaddingAwareAlignments for proper full-width behavior. Constrained vs flow layout types. Group block with layout settings. BAD: hardcoded max-width in CSS. GOOD: contentSize/wideSize in theme.json with useRootPaddingAwareAlignments.
9. **Custom CSS in theme.json** -- styles.css property (WP 6.2+) for additional CSS. Per-block custom CSS in styles.blocks.{block-name}.css. When to use: edge cases that theme.json properties can't express. BAD: large stylesheet in styles.css theme.json property. GOOD: minimal custom CSS for specific edge cases, bulk styling via theme.json properties.
10. **Block Locking in Templates** -- lock attribute on blocks in templates to prevent user editing. "lock": {"move": true, "remove": true}. Content-only editing via templateLock="contentOnly" on Group blocks. When to lock: structural elements (header layout, footer layout). When not to lock: content areas. BAD: locking all blocks (defeats FSE purpose). GOOD: locking structural blocks, leaving content editable.
11. **WordPress.org Theme Submission** -- Required files for block themes: style.css (with headers), templates/index.html, theme.json. Recommended: screenshot.png (1200x900), readme.txt. Theme Check plugin alignment. Common rejection reasons: hardcoded styles, missing required files, accessibility issues, missing text domain, GPL-incompatible assets. BAD: theme with hardcoded styles submitted to WordPress.org. GOOD: theme using theme.json for all styling.

All PHP examples: WordPress PHP Coding Standards. HTML examples: valid block markup. JSON examples: valid theme.json v3 syntax. BAD/GOOD pairs throughout. WP version markers for features.

**File 2: classic-to-block-guide.md (THM-27)**

Structure:
1. **Quick Reference Table** -- Migration mapping: Classic Feature | Block Theme Equivalent | Migration Complexity (low/medium/high). Cover: templates, header/footer, sidebars, menus, Customizer, add_theme_support, template tags, widgets, custom loops.
2. **Migration Strategy Overview** -- Three approaches: full conversion (rewrite as block theme), hybrid adoption (add theme.json incrementally), gradual migration (convert one piece at a time). Decision guide: when to use each. Small theme = full conversion. Large production theme = hybrid first.
3. **functions.php Mapping** -- Side-by-side table of common functions.php patterns and their theme.json equivalents:
   - add_theme_support('post-thumbnails') -> settings.blocks.core/post-featured-image
   - add_theme_support('custom-logo') -> site-logo block in templates
   - add_theme_support('title-tag') -> automatic in block themes
   - add_theme_support('editor-styles') -> styles in theme.json
   - add_theme_support('wp-block-styles') -> still valid (loads default block styles CSS)
   - register_nav_menus() -> wp:navigation block in templates
   - register_sidebar() -> template parts
   - Custom color palette via add_theme_support('editor-color-palette') -> settings.color.palette in theme.json
   - Custom font sizes via add_theme_support('editor-font-sizes') -> settings.typography.fontSizes in theme.json
   BAD: add_theme_support in block theme. GOOD: theme.json equivalent.
4. **Template Conversion** -- Step-by-step: header.php -> parts/header.html, footer.php -> parts/footer.html, sidebar.php -> parts/sidebar.html, single.php -> templates/single.html, page.php -> templates/page.html. PHP template tag to block markup mapping: the_title() -> wp:post-title, the_content() -> wp:post-content, the_excerpt() -> wp:post-excerpt, the_post_thumbnail() -> wp:post-featured-image, get_search_form() -> wp:search. The Loop -> wp:query + wp:post-template. Complete conversion example: classic single.php to block single.html.
5. **Sidebar to Template Part** -- register_sidebar() removed. Widget areas -> template parts with block content. dynamic_sidebar() calls replaced by template part references. Classic widgets have block equivalents (Recent Posts -> Latest Posts block, Categories -> Categories block, Search -> Search block). BAD: register_sidebar in block theme. GOOD: template part with blocks.
6. **Menu to Navigation Block** -- register_nav_menus() removed. wp_nav_menu() calls replaced by wp:navigation block in templates. Menu items -> navigation links. Submenus -> submenu blocks. Menu locations -> template part placement. Fallback: page list when no menu exists.
7. **Customizer to Global Styles** -- Customizer sections -> Site Editor > Styles. Custom Customizer controls -> not directly migrated (recommend removing). Color scheme Customizer -> theme.json color palette + style variations. Font Customizer -> theme.json typography + style variations. Background image Customizer -> limited in block themes (use cover block). BAD: Customizer in block theme. GOOD: style variations for user choices.
8. **Hybrid Theme Pattern** -- Incremental adoption: add theme.json to classic theme. What theme.json enables even without block templates: color palette in editor, font sizes in editor, layout settings, appearance tools. Next step: add one template part (header.html) while keeping classic templates. Eventually: convert all templates. BAD: all-or-nothing approach for large theme. GOOD: incremental migration preserving backward compatibility.
9. **Enqueue Patterns** -- Classic: wp_enqueue_style/wp_enqueue_script in functions.php. Block theme: style.css auto-enqueued, additional styles via wp_enqueue_block_style() for per-block conditional loading, scripts via viewScriptModule in block.json. Editor styles: add_editor_style() still works, theme.json styles also apply to editor. BAD: multiple unconditional wp_enqueue_style calls. GOOD: theme.json for most styles, wp_enqueue_block_style for per-block CSS.
10. **Common Migration Pitfalls** -- Table: forgetting to create templates/index.html (theme won't be recognized as block theme), leaving PHP template tags in .html files (won't execute), missing template part registration in theme.json, not converting conditional logic (use template hierarchy instead), keeping register_sidebar() in block theme (no widget areas), hardcoding menus in templates instead of using Navigation block, not updating style.css to minimal content for block theme, forgetting to add "Requires at least" and "Tested up to" headers.

PHP examples: WordPress PHP Coding Standards. HTML block markup examples: valid syntax. BAD/GOOD pairs showing classic vs block approach throughout. Cross-reference wp-block-development for block markup details. Cross-reference wp-security-review for escaping in classic templates being migrated.
