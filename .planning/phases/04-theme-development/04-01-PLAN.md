**12 Sections (mirror block/plugin/security skill layout):**

1. **Overview** -- Systematic theme development review for WordPress 6.6+. Core principle: WordPress themes have undergone a paradigm shift -- block themes use HTML templates with block markup and theme.json for global settings/styles, replacing PHP templates and add_theme_support() calls. theme.json v3 is the single source of truth for design configuration. Review validates theme.json structure, template hierarchy, template parts, global styles, style variations, block patterns, and functions.php setup. Auto-detects theme type (block/classic/hybrid/child) and adjusts guidance accordingly. Report with line numbers, severity, BAD/GOOD code pairs. Note: This skill reviews PHP, HTML template, JSON, and pattern files.

2. **When to Use / Don't Use** -- Use for: block theme structure review, theme.json v3 validation, template hierarchy audit (block + classic), template parts review, global styles analysis, style variations validation, block patterns in themes, child theme compatibility check, classic-to-block migration assessment, WordPress.org theme submission readiness. Don't use for: block editor component review (use wp-block-development), plugin architecture audits (use wp-plugin-development), security-only audits (use wp-security-review), performance-only audits (use wp-performance-review), WooCommerce theme integration (use wp-woocommerce-development when available), visual design assessment (code review only).

3. **Code Review Workflow** -- 7-step workflow:
   1. Detect theme type from file structure:
      - Block theme: has templates/index.html + theme.json + style.css
      - Classic theme: has index.php + style.css + functions.php, no templates/ directory
      - Hybrid theme: has index.php + theme.json (classic with block theme features)
      - Child theme: has style.css with Template: parent-slug header
      - WordPress.org submission: check for required files per theme type
   2. Validate theme.json (if present):
      - Version must be 3 (WP 6.6+). Flag v2 as WARNING with upgrade steps. Flag v1 as CRITICAL.
      - Settings: validate color palettes, typography scales, spacing, layout (contentSize/wideSize), border, shadow, dimensions, position
      - Styles: validate root styles, element styles (heading/link/button), per-block style overrides
      - templateParts: validate name, title, area (header/footer/uncategorized)
      - customTemplates: validate name, title, postTypes array
      - patterns: validate slugs array (WordPress.org pattern directory)
   3. Check template hierarchy:
      - Block themes: templates/ directory with .html files (index.html required, plus single.html, page.html, archive.html, 404.html, search.html, home.html, front-page.html)
      - Classic themes: PHP template files following WordPress hierarchy
      - Verify required templates exist per theme type
   4. Check template parts (block themes):
      - parts/ directory with header.html, footer.html, sidebar.html
      - Area designations in theme.json templateParts
      - Proper block markup in part files
   5. Scan for CRITICAL patterns:
      - Missing required files (templates/index.html for block themes, index.php for classic)
      - Invalid or missing theme.json version
      - Missing style.css with required headers (Theme Name, Version minimum)
      - Template hierarchy violations (wrong filenames, wrong directory)
      - theme.json v1 (deprecated)
   6. Check WARNING patterns:
      - Hardcoded styles in block theme templates (inline style attributes, style tags)
      - theme.json v2 instead of v3
      - Missing useRootPaddingAwareAlignments in theme.json layout settings
      - Classic patterns used where block patterns preferred (add_theme_support vs theme.json)
      - Missing style variations support
      - Block patterns without proper PHP headers in patterns/ directory
      - Child theme overriding parent functions without checking function_exists()
   7. Note INFO improvements:
      - Could use theme.json settings instead of add_theme_support()
      - Missing custom template registration in theme.json customTemplates
      - Style variation naming conventions
      - Font could use local hosting instead of Google Fonts CDN
      - Missing index.php safety files in subdirectories
   Report using output format below. If security concerns found, suggest /wp-sec-review. If plugin architecture issues found, suggest /wp-plugin-review. If block markup issues found, suggest /wp-block-review.

   **Theme Type Detection:** Auto-detect theme type FIRST, then apply appropriate review path. Block themes get FSE-focused review. Classic themes get migration suggestions. Hybrid themes get incremental adoption guidance. Child themes get override compatibility checks.

4. **File-Type Specific Checks** -- Organized by file type:

   **theme.json (THM-02, THM-10, THM-13):**
   - Version must be 3 (WP 6.6+). Flag 2 as WARNING with v2 to v3 migration. Flag 1 as CRITICAL.
   - v3 breaking changes: defaultFontSizes and defaultSpacingSizes both default to true (reversed from v2)
   - settings: color (palette, gradients, duotone, custom, defaultPalette), typography (fontSizes, fontFamilies, customFontSize, fontWeight, letterSpacing, textDecoration, fluid), spacing (padding, margin, blockGap, units, customSpacingSize), layout (contentSize, wideSize), border (color, radius, style, width), shadow (presets, defaultPresets), dimensions (aspectRatio, minHeight), position (sticky)
   - styles: root-level (color, typography, spacing, border), elements (heading, link, button, caption, cite), blocks (per-block overrides with full style support)
   - templateParts: array of {name, title, area} -- area must be header, footer, or uncategorized
   - customTemplates: array of {name, title, postTypes} -- postTypes array
   - patterns: array of WordPress.org pattern directory slugs
   - $schema field recommended for IDE validation
   - useRootPaddingAwareAlignments in settings.layout (THM-20): MUST be true for proper full-width alignment

   **Template files -- block themes (THM-03, THM-05, THM-17):**
   - templates/ directory: .html files using block markup (wp:group, wp:template-part, wp:post-title, wp:post-content, wp:query, etc.)
   - Required: templates/index.html (fallback template)
   - Standard hierarchy: front-page.html, home.html, single.html, page.html, archive.html, 404.html, search.html, category.html, tag.html, author.html
   - Custom templates registered via customTemplates in theme.json
   - Templates reference parts via wp:template-part block with slug attribute

   **Template parts -- block themes (THM-04, THM-18):**
   - parts/ directory: header.html, footer.html, sidebar.html
   - Each part must be registered in theme.json templateParts with area
   - Areas: header, footer, uncategorized (sidebar uses uncategorized or general)
   - Block markup inside parts: Navigation block, Site Title, Site Logo, etc.

   **Template files -- classic themes (THM-03, THM-06, THM-07):**
   - PHP template hierarchy: index.php, single.php, page.php, archive.php, 404.php, search.php, front-page.php, home.php
   - get_template_part() for modular template loading
   - Conditional tags: is_front_page(), is_singular(), is_page(), is_single(), is_archive(), is_category(), is_home()
   - Template tags: the_title(), the_content(), the_excerpt(), the_permalink(), etc.
   - Cross-reference wp-security-review for escaping in template output

   **style.css:**
   - Required headers: Theme Name, Version (minimum for WordPress.org)
   - Block themes: style.css is primarily metadata, actual styling via theme.json
   - Classic themes: style.css contains primary styles
   - Child themes: Template header required pointing to parent slug

   **functions.php (THM-08, THM-09, THM-16):**
   - Block themes: minimal setup -- enqueue styles, register block patterns, register custom block styles
   - Classic themes: add_theme_support() calls, register_nav_menus(), register_sidebar(), enqueue scripts/styles
   - Both: after_setup_theme hook for setup, wp_enqueue_scripts for assets
   - Child theme safety: check function_exists() before overriding parent functions
   - Cross-reference wp-plugin-development for hook patterns and functions.php best practices

   **Block patterns in themes (THM-11):**
   - patterns/ directory with PHP files
   - Required headers: Title, Slug, Categories (optional: Description, Keywords, Block Types, Post Types, Inserter)
   - Auto-registered by WordPress when in patterns/ directory
   - BAD: register_block_pattern() in functions.php for theme patterns. GOOD: PHP files in patterns/ directory.
   - Cross-reference wp-block-development for block pattern content

   **Style variations (THM-19):**
   - styles/ directory with alternate theme.json files
   - Each file overrides settings and/or styles from root theme.json
   - Must have title property at root level
   - Naming convention: descriptive slug matching the visual style

   **Global styles (THM-10):**
   - Color palettes: settings.color.palette array with slug/color/name
   - Typography: settings.typography.fontSizes, fontFamilies with font-face declarations
   - Spacing: settings.spacing.spacingSizes for custom scale, spacingScale for generated scale
   - Layout: settings.layout.contentSize and wideSize
   - Fluid typography: settings.typography.fluid for responsive font sizes

   **Surface-level checks:**
   - Navigation block configuration (wp:navigation in templates, nav menu fallback)
   - Custom template registration (customTemplates in theme.json)
   - Font management (fontFamilies in theme.json, local font hosting in assets/)
   - theme.json v2 vs v3 differences (defaultFontSizes, defaultSpacingSizes)
   - add_theme_support() calls still valid in block themes (wp-block-styles, responsive-embeds, editor-styles)
   - Classic conditional tags (is_front_page, is_singular, etc.)
   - Classic get_template_part() patterns
   - Widget areas (classic themes only -- block themes use template parts)
   - Block locking in templates (lock attribute for preventing user editing)
   - Starter theme patterns (Underscores-based themes, common boilerplate)

   Each check item includes: pattern to find, severity level, brief explanation.

10. **Common Mistakes (THM-24)** -- Table of patterns that look like issues but are NOT:
    - Block theme style.css with minimal CSS is correct (styling via theme.json, not CSS)
    - functions.php being small or absent in block themes is fine (theme.json handles most configuration)
    - No sidebar.php in block themes (use template parts instead)
    - No header.php/footer.php in block themes (use parts/header.html and parts/footer.html)
    - Empty or absent customTemplates in theme.json (only needed for non-standard templates)
    - patterns array in theme.json with WordPress.org slugs (these are remote patterns, not local)
    - No register_nav_menus() in block themes (Navigation block handles menus)
    - No register_sidebar() in block themes (template parts replace widget areas)
    - theme.json without templateParts (WordPress auto-discovers parts/ directory files)
    - add_theme_support( 'wp-block-styles' ) in block themes is still valid (loads default block styles)
    - Block template files without PHP (they use block markup, not PHP template tags)
    - Theme without screenshot.png (recommended but not required for development)
